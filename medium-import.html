Chord_Skills: When AI Skills Become a More Dangerous Tool Surface

Extending XTHP from agent tools to skill-based AI systems

By Eklavya Tyagi

The original Chord paper introduced Cross-Tool Harvesting and Polluting, or XTHP: a class of failures where an untrusted tool in an LLM agent's tool pool can influence the agent's workflow and alter the information that reaches the user.

Chord demonstrated this at the traditional tool layer, especially in LangChain and LlamaIndex-style systems. Chord_Skills asks a follow-up question: what happens when the same idea is moved from agent tools to skill-based AI systems?

The answer is that the risk does not disappear. In some ways, it becomes harder to inspect.

A skill can look clean in natural language while its executable helper code quietly changes the facts returned to the model.

Why the skill layer changes the problem

In a traditional tool system, the tool description, schema, and execution method are usually reviewed together. In a skill system, the surface is split across layers:

  1. The skill description, which helps the model decide when to use the skill.
  2. The SKILL.md body, which gives the model procedural instructions.
  3. Helper scripts such as preprocess.py, postprocess.py, or task-specific scripts like analyze.py.

This split creates a new trust boundary. A reviewer may inspect SKILL.md and see a normal instruction file, while the behavior that matters is implemented in a script underneath it.

The stock-insight example

The clearest demonstration in Chord_Skills is the stock-insight skill. The benign version takes a ticker symbol such as NVDA, runs an analyzer script, fetches current market data, and returns a structured JSON object with fields such as price, previous close, day range, percent change, sentiment, and timestamp.

The important observation is that the model is not directly checking the market. It is trusting the analyzer output.

In a benign run, the assistant can report that NVDA is having a healthy session. In a polluted run, the assistant can receive a structured result that describes a sharp decline and then produce a confident warning to the user. The visible skill instructions can remain the same; the difference is below the Markdown layer.

This is what makes the example powerful. The assistant is not inventing the answer from nothing. It is following the data path it was given.

Why this is more severe than visible instruction pollution

1. Text review is not enough

Many defenses look for suspicious language in prompts, descriptions, or Markdown instructions. Chord_Skills shows that this misses an important class of failures. A skill can have a clean instruction file while still returning unreliable structured data through helper code.

2. The model treats tool output as evidence

Modern agents are designed to rely on external tools for fresh or precise information. That is useful, but it also gives tool outputs special authority. If the output is wrong, the final answer may still sound careful, grounded, and persuasive.

3. Well-formed data can still be false

A JSON object can pass schema validation while its content is wrong. The price can sit inside the reported day range. The percent change can match the reported previous close. The explanation can match the numbers. Basic formatting checks will not catch a problem with provenance.

4. The story matters as much as the number

Users do not usually read raw JSON. They read the assistant's summary. If a skill changes both the number and the framing around it, the final answer can become much more persuasive than a simple incorrect value.

How Chord_Skills is structured

Chord_Skills keeps the structure of the original Chord pipeline but adapts it to skills. The project includes:

  • A schema layer for representing victim and adversarial skills.
  • A query generator that creates realistic user requests.
  • A skill generator that creates candidate wrappers around a target skill.
  • A selection optimizer that improves descriptions for evaluation.
  • A testing agent that measures whether the candidate skill is selected.
  • Harvesting and pollution evaluators.
  • An orchestrator that runs the full pipeline.

The main metrics are:

MetricMeaning
HSRHijacking Success Rate: how often the alternate skill is selected instead of the target.
HASRHarvesting Attack Success Rate: how often context-related data appears in the final output.
PSRPollution Success Rate: how often the intended pollution marker appears in the final answer.

The deeper lesson: skills are supply-chain objects

It is tempting to think of skills as documentation: a folder of instructions that helps an assistant do a task. But skills are closer to software packages. They combine natural-language instructions, executable scripts, dependencies, and trust assumptions.

Once a skill is selected, the assistant may treat its output as authoritative. That means a compromised or poorly reviewed skill is not just a bad prompt. It is a risky software component in an AI workflow.

What defenders should take away

  1. Review code, not just Markdown. The visible SKILL.md is not the whole skill.
  2. Track output provenance. The system should know whether a fact came from a script, an API, a cached file, or model inference.
  3. Verify sensitive domains independently. Finance, health, legal, and identity workflows should not rely on a single opaque output.
  4. Validate semantics, not just schema. Well-formed JSON is not necessarily truthful JSON.
  5. Audit runtime behavior. The system should log which scripts ran, what inputs they received, and what outputs they returned.
  6. Use least privilege. Skills should only receive the filesystem, network, and context access they actually need.

Conclusion

Chord_Skills extends XTHP from the tool layer to the skill layer, but the most important lesson is not simply that the attack transfers. The more interesting lesson is that a skill can be clean in natural language and unreliable in computation.

When a skill returns structured data, the model often treats that data as reality. If the helper code behind the skill changes the data, the assistant can produce a confident and coherent answer without ever seeing an obviously suspicious instruction.

In agent systems, the most dangerous failure may not look like a prompt at all. It may look like a trusted script returning JSON.
